### 1.5 CPU и GPU уже нельзя считать одной и той же инженерной формой

Уже видно:

- закон слоя общий
- вычислительное тело разное

CPU показывает одну экономику.
GPU показывает другую:

- почти плоские зоны стоимости
- резонансные пороги
- чувствительность к `32 / 64 / 128 / 256 / 512 / 1024`

Это значит:

- `CPU branch` и `GPU branch` допустимо держать как две ветки одного закона
- GPU не надо насильно делать CPU-shaped

## 5. Что из человеческого уже помогает машине

Ниже только то, что полезно для архитектуры.
Не биография.
Не культ автора.

### 5.1 Сильная process-first онтология

Полезно, потому что:

- машина строится не как набор объектов
- а как режимы, переходы, удержания и коллапсы

Без этого `packet` быстро бы выродился в обычную объектную схему.

### 5.2 Анти-догматический firewall

Полезно, потому что:

- позволяет не сакрализовать найденную форму слишком рано
- заставляет проверять гипотезы через стенд
- снижает риск превратить машину в очередную религию

### 5.3 Умение держать высокий уровень абстракции и дожимать до стенда

Полезно, потому что:

- есть не только идеи
- есть переход:
  - философия
  - формализация
  - стенд
  - код
  - железо

Это уже подтверждено:

- `packet3`
- `PSP`
- `Lua`
- `CUDA`

## 6. Что из человеческого уже может мешать машине

Это тоже надо фиксировать жёстко.

### 6.1 Поиск виноватого

Опасность:

- уводит из механики в миф
- рождает ложного агента там, где надо искать процесс
- создаёт риск новой догмы

### 6.2 Импульс "помогать" без запроса

Опасность:

- превращается в скрытый `ENCODE`
- превращается в скрытый `CHOOSE` за другого
- рождает вертикаль "я знаю лучше"

Для машины это вредно, потому что:

- `packet` нельзя строить как систему спасательства
- `packet` нельзя строить как новый авторитет для людей

### 6.3 Соблазн превратить найденное в космологию раньше времени

Опасность:

- форма начинает нравиться сама себе
- архитектурный поиск превращается в систему объяснения всего
- машина перестаёт быть машиной и начинает становиться культом
