# Варианты Crazy Torus

Этот документ нужен для одного:

- накидать несколько реализационных гипотез `crazy-step`

Не код.
Не обучение.
Не полный `packet`.

Только варианты того,
как может жить первый `L1 chaos` в виде `crazy-torus`.

## Общая рамка

У всех вариантов ниже общее:

- геометрия = `1D ring`
- есть узлы
- у узла есть `core`
- у узла есть `trace`
- есть одна бегущая волна
- волна имеет carry-state `A`
- след должен перезаписываться

Различие только в одном:

- как именно делается `crazy-step`

## Вариант A. Буквальный перенос

Это самый тупой и самый честный вариант.

### Закон

На каждом шаге:

1. волна стоит на узле `p`
2. берёт соседний узел `q`
3. читает `A`
4. строит operand = `core[q]`
5. считает `res = crazy(A, operand)`
6. пишет:
   - `A = res`
   - `core[p] = res`
   - `trace[p] = crazy(trace[p], res)`

После этого сдвигается на следующий узел.

### Плюсы

- максимально близко к исходной физике
- почти не фантазируем
- легко понять, что именно происходит

### Минусы

- может получиться слишком жёстко
- `core` может слишком быстро переписываться
- след может не дать новой динамики, а только дублировать `core`

### Риск

Система может слишком быстро схлопнуться в однообразный проход.

## Вариант B. Core Through Trace

Здесь мы делаем важную поправку:

- волна работает не только с `core`, но и с уже оставленным следом

### Закон

На каждом шаге:

1. волна стоит на `p`
2. читает `core[p]`
3. читает `trace[p]`
4. строит operand = `crazy(core[p], trace[p])`
5. считает `res = crazy(A, operand)`
6. пишет:
   - `A = res`
   - `core[p] = crazy(core[p], res)`
   - `trace[p] = res`

### Плюсы

- след реально начинает участвовать в исполнении
- программа и след сцепляются сильнее
- второй круг уже точно идёт по другому полю

### Минусы

- система может стать слишком нестабильной
- след может начать доминировать слишком рано

### Риск

Быстрый уход в мусор.

## Вариант C. Position-Biased Crazy

Здесь мы сохраняем важнейшее свойство Malbolge:

- смысл зависит от позиции

### Закон

На каждом шаге:

1. волна стоит на `p`
2. берёт `phase[p]`
3. строит position-bias = функция от `p mod k`
4. строит operand = `crazy(core[p], position-bias)`
5. считает `res = crazy(A, operand)`
6. пишет:
   - `A = res`
   - `core[p] = crazy(res, trace[p])`
   - `trace[p] = crazy(trace[p], phase[p])`

### Плюсы

- позиционность встроена сразу
- одно и то же состояние в разных местах ведёт себя по-разному
- ближе к malbolge-like смыслу

### Минусы

- сложнее анализировать
- если bias выбрать плохо, всё станет искусственным

### Риск

Позиция начнёт определять всё,
и собственная жизнь поля ослабнет.

## Вариант D. Dual Neighbor Wave

Здесь волна читает не одного соседа, а два направления сразу.

### Закон

На каждом шаге:

1. волна стоит на `p`
2. читает `left = core[p-1]`
3. читает `right = core[p+1]`
4. строит operand = `crazy(left, right)`
5. считает `res = crazy(A, operand)`
6. пишет:
   - `A = res`
   - `core[p] = crazy(core[p], res)`
   - `trace[p] = crazy(trace[p], operand)`

### Плюсы

- поле чувствует локальную форму, а не только линию прохода
- это уже похоже на настоящее графовое соседство
- больше шансов на сложную жизнь

### Минусы

- это уже шаг дальше от буквального Malbolge
- сложнее читать динамику

### Риск

Слишком рано вырастет сложность,
и мы не поймём, жив ли принцип сам по себе.

## Предварительное сравнение

Если идти строго по нашей методологии,
то стартовая приоритетность такая:

1. Вариант A
2. Вариант B
3. Вариант C
4. Вариант D

### Почему A первым

Потому что:

- он самый тупой
- он самый честный
- он минимально фантазирует
- если даже он не даёт жизни, это важный сигнал

### Почему B вторым

Потому что он даёт trace реальную роль,
но ещё не взрывает систему слишком большим числом новых сущностей.

## Что считаем хорошим первым выбором

На текущем этапе наиболее разумный первый кандидат:

- Вариант A как baseline

И сразу после него, если baseline мёртв или слишком туп:

- Вариант B

## Чего не делать сейчас

Если v0 жив,
всё это успеем.
