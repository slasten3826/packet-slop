# Packet-Slop :: L2 Four-Node Boundary Prototype

Черновой документ второго слоя.

Это не финальный канон.
Это первая чистая сборка того, как `L2` сейчас реально читается из:

- нового понимания слоёв
- старого `zig`-прототипа
- текущей packet-онтологии

## 1. Основная идея

`L2` не должен тащить все 8 операторов.

На текущем этапе для него достаточно 4 узлов:

- `OBSERVE`
- `RUNTIME`
- `CHOOSE`
- `ENCODE`

Это не "урезание ради простоты".
Это честное выделение того, что реально принадлежит второму слою.

## 2. Позиция слоя в дереве

Слои сейчас читаются так:

- `L1 = FLOW + DISSOLVE + CONNECT`
- `L2 = OBSERVE + RUNTIME + CHOOSE + ENCODE`
- `L3 = CALM + MANIFEST-body`

То есть:

- `FLOW` уже живёт как `CHAOS` в `L1`
- `DISSOLVE` и `CONNECT` тоже уже принадлежат `L1`
- `MANIFEST` не должен сидеть внутри `L2`
- `L2` — это boundary-мембрана между хаосом и кристаллом

## 3. Ось второго слоя

`L2` строится не как плоская равноправная сетка,
а как слой между двумя полюсами:

- `OBSERVE` смотрит вниз, в `L1`
- `RUNTIME` смотрит вверх, в `L3`

Между ними работают:

- `CHOOSE`
- `ENCODE`

То есть второй слой — это не просто "операторы на карте",
а ось:

```txt
L1 / CHAOS
   ↓
OBSERVE
   ↓
CHOOSE ↔ ENCODE
   ↓
RUNTIME
   ↓
L3 / CALM
```

Это не обязательно одна линия в физическом теле,
но семантически слой сейчас выглядит именно так.

## 4. Семантика четырёх узлов

### OBSERVE

Нижний глаз слоя.

Задачи:

- читать давление из `L1`
- читать fingerprint / turbulence / saturation
- различать живую турбулентность и застой
- поднимать наверх не сырой шум, а локальное чтение среды

`OBSERVE` не кристаллизует и не выбирает.
Он только делает среду видимой для `L2`.

### RUNTIME

Верхний глаз слоя.

Задачи:

- смотреть в `L3`
- знать, насколько `CALM` уже насыщен
- видеть ресурсное и структурное состояние верхнего слоя
- удерживать временное окно, чтобы маршрут не распался раньше времени

`RUNTIME` не пишет программу и не делает manifest.
Он удерживает контур между `CHOOSE/ENCODE` и `CALM`.

### CHOOSE

Схлопывающий центр.

Задачи:

- брать несколько возможных продолжений
- выбирать одно
- повышать устойчивость выбранного маршрута
- отбрасывать лишние продолжения

Это не просто switch.
Это локальный акт коллапса.

### ENCODE

Кристаллизующий центр.

Задачи:

- брать устойчивый маршрут после `CHOOSE`
- кодировать его в форму, пригодную для `L3`
- делать это с потерей
- переносить уже не жизнь, а кристалл

`ENCODE` не наблюдает и не держит runtime.
Он делает перевод между boundary-динамикой и calm-телом.

## 5. Связь с zig prototype

Старый `zig L2` уже частично содержал эту архитектуру:

- `ObserveA` читал `CHAOS`
- `ObserveB` читал `CALM`
- `Choose` принимал решение
- `Encode` кристаллизовал в триграммы

Но это было собрано как:

- два наблюдателя
- один chooser
- один encoder

Новая гипотеза делает это яснее:

- `ObserveA` по сути = `OBSERVE`
- `ObserveB` по сути = `RUNTIME-side observation`
- `Choose` = `CHOOSE`
- `Encode` = `ENCODE`

То есть новый `L2` не ломает `zig`,
а выделяет его скрытую структуру более честно.

## 6. Форма тела

Физическое тело `L2` сейчас видится не как линия и не как обычный граф,
а как:

- тороидальная сетка
- скорее `hex`, чем квадрат
- с 4 семантическими типами узлов
- с локальными связями
- с несколькими ходоками / фронтами исполнения

Но важный момент:
сетка здесь не самоцель.
Сетка — это только тело для четырёх узлов.

## 7. Минимальная клетка

Клетка `L2` может быть минимально такой:

```txt
Cell {
    kind: NodeKind,      // OBSERVE | RUNTIME | CHOOSE | ENCODE
    charge: u8,          // локальное возбуждение
    phase: u8,           // локальная фаза
    links: LinkMask,     // допустимые направления
    mark: u8,            // временная метка
}
```

Где `NodeKind`:

```txt
enum NodeKind {
    Observe,
    Runtime,
    Choose,
    Encode,
}
```

## 8. Минимальный цикл тика

Черновой тик второго слоя:

```txt
1. read_chaos_into_observe_nodes()
2. read_calm_into_runtime_nodes()
3. propagate_charge_over_links()
4. choose_nodes_collapse_candidates()
5. encode_nodes_mark_stable_routes()
6. runtime_nodes_hold_or_release_routes()
7. emit_crystal_to_L3_if_ready()
```

## 9. Что значит "смотреть в обе стороны"

Это не означает, что `L2` — просто двусторонняя шина.

Это означает:

- нижняя сторона слоя знает о состоянии `CHAOS`
- верхняя сторона слоя знает о состоянии `CALM`
- середина слоя (`CHOOSE + ENCODE`) живёт в напряжении между ними

То есть второй слой — это реально мембрана,
а не однонаправленный pipeline.

## 10. Почему здесь не нужны все 8 операторов

Потому что оставшиеся операторы уже принадлежат другим слоям:

- `FLOW` — `L1`
- `DISSOLVE` — `L1`
- `CONNECT` — `L1`
- `MANIFEST` — `L3`

А `LOGIC` и `CYCLE` на текущем этапе можно считать:

- либо внутренними режимами узлов
- либо поздними добавками
- либо производными паттернами, а не базовыми узлами `L2`

То есть four-node architecture сейчас выглядит честнее, чем full-8 grid.

## 11. Самый короткий тезис

`L2` сейчас читается как:

> четырёхузловая boundary-мембрана между `CHAOS` и `CALM`,
> где `OBSERVE` смотрит в `L1`,
> `RUNTIME` смотрит в `L3`,
> а `CHOOSE` и `ENCODE` делают коллапс и кристаллизацию между ними.

## 12. Следующий шаг

После этого документа следующий разумный шаг уже не философский, а инженерный:

- сделать минимальный `L2 stand`
- взять `hex`-тор
- завести 4 типа узлов
- подать synthetic pressure снизу
- проверить, возникают ли устойчивые маршруты между `OBSERVE` и `RUNTIME`
- и только потом думать про более богатую топологию
