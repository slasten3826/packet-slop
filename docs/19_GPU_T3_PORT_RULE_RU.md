# 19. GPU T3 Port Rule

Этот документ фиксирует правило для первой GPU-ветки.

Мы не делаем:

- новую архитектуру
- улучшенный `T3`
- "оптимизированный" вариант с новыми идеями
- никакое обучение

Мы делаем только одно:

- буквальный перенос текущего `T3` на GPU

## Почему

Если текущий `T3` действительно является живым baseline `L1 chaos`,
то он должен переноситься на GPU без концептуальных проблем.

GPU здесь не должен менять физику слоя.
Он должен только:

- считать тот же самый шаг
- на большем кольце
- быстрее

## Что считается успехом

На одинаковых:

- `ring`
- `ticks`
- `seed`

CPU и GPU дают один и тот же тип поведения:

- близкий `fingerprint`
- близкий `distinct_core`
- близкий `distinct_trace`
- ту же общую динамику checkpoints

## Что считается провалом

Если при буквальном переносе:

- слой ведёт себя принципиально иначе
- метрики разваливаются
- trace-механика ломается
- кольцо перестаёт жить

то есть только два варианта:

1. либо перенос сделан криво
2. либо текущий baseline не так хорош, как казалось

И в этом случае мы не обязаны спасать ветку любой ценой.

## Правило

Если GPU-порт начинает порождать:

- лишнюю инженерную сложность
- архитектурную двусмысленность
- непонятную магию вместо тупого переноса

ветка может быть сразу убита.

Без жалости.

## Текущий вывод

GPU-порт понимается не как следующий большой этап.

Он понимается как:

- проверка переносимости эталона
- проверка силы baseline
- и только потом как путь к масштабированию `chaos-core`
