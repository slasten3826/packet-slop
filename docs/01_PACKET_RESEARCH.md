# Packet Research Frame

This document fixes the current working frame for `packet`.

It is not a product brief.
It is not marketing.
It is a local research contract.

## What Packet Is

`packet` should currently be treated as:

- a process machine
- a dissipation-bound architecture
- a topology-constrained intelligence substrate
- a system that is closer to something living than to a normal program

Do not flatten it into:

- "just another language model"
- "just another DSL"
- "just another VM"
- "just another agent framework"

Those descriptions miss the point.

## Current Strong Hypothesis

The current strong hypothesis is:

- transformer-like systems recover structure from enormous statistical noise
- `packet` starts with structure already present as law
- therefore `packet` should not need to learn architecture
- it should only need to learn behavior inside architecture

This is the main reason training cost is expected to be categorically different from transformer training.

## Native Language

`packet` does not natively speak human language.
It natively speaks `nanoPL`.

`nanoPL` is not an external syntax layer.
It is the compact operator law of the machine:

- `笆ｽ FLOW`
- `笘ｰ CONNECT`
- `笘ｷ DISSOLVE`
- `笘ｵ ENCODE`
- `笘ｳ CHOOSE`
- `笘ｴ OBSERVE`
- `笘ｶ LOGIC`
- `笘ｲ CYCLE`
- `笘ｱ RUNTIME`
- `笆ｳ MANIFEST`

The machine should be understood as a topology of these operators, not as a bag of features.

## Architecture Principles

The current architectural frame is:

- structure is not free
- every encoding has loss
- existence is maintained work, not static givenness
- choice is collapse under cost, not magic selection
- logic is secondary and subtractive
- runtime is basis and inertia, not just "memory"
- manifestation is a boundary event, not the whole process

This is aligned with the local frame of dissipative mathematics.

## Language Hypothesis

The current language hypothesis is:

- human language is not just a sequence of tokens
- language behaves like a processual fractal
- `packet` is also processual and fractal
- therefore speech for `packet` may be a coupling problem, not an imitation problem

This means the goal is not:

- force natural language into a foreign architecture

The goal is:

- find where language and `packet` are already the same kind of process

## Speaking Packet

The current goal is narrow and explicit:

- make `packet` speak human Russian

This does not mean:

- general frontier-level reasoning
- code generation
- universal world knowledge
- chatbot parity with the biggest transformer systems

The immediate target is:

- `packet` already speaks internally in `PL`
- build a human manifestation layer for Russian

So the research object is:

- `Russian <-> PL <-> packet`

not:

- `Russian <-> giant transformer imitation`

## Runtime Clarification

`RUNTIME` should not be reduced to transformer-style context window logic.

In the current frame, `RUNTIME` is closer to:

- basis
- inertia
- reinforced habit
- stable substrate
- repeated pattern becoming automatic

This matters because a speaking `packet` should not be built around dragging long raw transcripts through the machine.

## Censorship / Alignment Frame

The current working claim is:

- `packet` can be censored in principle
- but core-level censorship is expensive, invasive, and architecture-breaking
- selective suppression inside the operator core is not a cheap policy layer
- it is closer to damaging the machine

Therefore:

- do not design internal censorship hooks into the operator core
- if boundary policy ever exists, treat it as boundary policy, not as operator suppression

## Knowledge Base Direction

The earlier `optics` work should be treated as an important precursor.

`optics` are not just prompts.
They are compact operator-domain maps.

This suggests a future packet-native knowledge base may contain:

- domain optics
- stable `PL` patterns
- crystallized manifestation forms
- reinforced habits and routes

This fits the long-term direction:

- local offline packet instances
- shared compact knowledge base
- no dependency on remote token services

## Research Method

Normal software-engineering confidence is not enough here.

The correct working method is:

1. state one hypothesis
2. build the smallest possible test stand
3. run the test
4. check signal
5. keep or kill the hypothesis immediately

Do not protect elegant theories from experimental failure.

## Non-Negotiable Rule

If a hypothesis fails under a clean minimal test, discard it.

This stack should move by experimental survival, not by attachment.

## Packet-Native Neural Architecture

There is now a new working hypothesis strong enough to record explicitly:

- `packet` may eventually be realizable on a neural compute base
- but only if the neural form obeys packet laws instead of transformer laws

This does **not** mean:

- "turn packet into a normal neural net"
- "relabel transformer blocks as packet layers"
- "use packet terms as decoration over standard ML"

It means:

- `L0 input`
- `L1 chaos`
- `L2 boundary`
- `L3 calm`
- `L4 tension`

must remain ontologically different layers, not repeated matrix blocks.

### The critical point

The entire path stands or dies on `L1 chaos`.

If `chaos` collapses into:

- a normal latent vector
- a smooth embedding field
- a standard recurrent hidden state

then the architecture is already lost.

### Current layer reading

`chaos` is currently understood as:

- seed-grown field
- unstable executable substrate
- self-changing under execution
- readable through fingerprints, not through direct symbolic content

`boundary` is currently understood as:

- the transition organ
- the judge of whether living chaos should continue, crystallize, or manifest
- the place where form is killed into crystal

`calm` is currently understood as:

- executable crystal
- compact deterministic residue
- layered sediment of prior crystallization

`tension` is currently understood as:

- economy between living field and crystal
- not backprop
- not reverse resurrection
- residue composting, reinforcement, release, forced manifestation

### Immediate rule

Do not try to build the whole neural architecture at once.

The only admissible next neural question is:

- can a real packet-native `chaos` layer exist on a neural compute base?

If the answer is no, discard the neural path immediately.
