# Packet-Slop :: L2 Prototype

Черновой прототип второго слоя.

Это не финальная спецификация.
Это первый технический эскиз того, как `BOUNDARY` может получить собственное тело.

## 1. Позиция слоя

- `L1 = CHAOS`
- `L2 = BOUNDARY GRID`
- `L3 = CALM`

`L2` не содержит:

- `FLOW`
- `MANIFEST`

Потому что:

- `FLOW` уже ушёл в `L1` как живая хаотическая среда
- `MANIFEST` должен жить в `L3` как кристаллизованная проявленность

Значит `L2` — это не ещё один хаос-слой и не язык вывода.
Это промежуточная операторная мембрана.

## 2. Основная гипотеза

`L2` мыслится как тороидальная потенциальная сетка.

Сетка:

- замкнута как тор
- расширяется влево/вправо/вверх/вниз
- существует как поле возможных связей
- активируется давлением из `L1`
- схлопывает устойчивые маршруты в `L3`

Это ближе не к обычному `Befunge`, а к:

- клеточному автомату
- с пространственным исполнением
- с локальной топологической записью
- по законам операторной онтологии `PL`

## 3. Операторы L2

Внутри `L2` остаются 8 операторов:

- `CONNECT`
- `OBSERVE`
- `DISSOLVE`
- `ENCODE`
- `CHOOSE`
- `LOGIC`
- `RUNTIME`
- `CYCLE`

Это согласуется со старым `zig`-телом `L3`, где триграммы уже mapped на эти 8 операторов.

## 4. Что такое клетка

Клетка не хранит "кусок программы" в обычном смысле.

Минимальное состояние клетки:

```txt
Cell {
    op: OperatorId,        // 1 из 8 операторов
    charge: u8,            // локальное возбуждение / давление
    phase: u8,             // локальная фаза цикла
    links: LinkMask,       // разрешённые направления связи
    mark: u8,              // служебная локальная метка
    rewrite: u8,           // pending local rewrite
}
```

Где:

- `op` — идентификатор операторного узла
- `charge` — насколько клетка сейчас активна
- `phase` — локальный шаг цикла
- `links` — разрешённые связи с соседями
- `mark` — временная локальная память
- `rewrite` — отложенная локальная перестройка

## 5. Тор и соседство

Базовое соседство:

- `N`
- `S`
- `W`
- `E`

Тороидальность:

- выход за правую границу возвращает в левую
- выход за нижнюю возвращает в верхнюю

Диагонали можно пока не вводить.
Сначала достаточно креста из 4 соседей.

## 6. Игроки / ходоки

В обычном `Befunge` ходок один.

Здесь как минимум 4 активных ходока:

- северный
- южный
- западный
- восточный

Это не обязательно "персонажи".
Это могут быть 4 фронта активации или 4 направленных луча исполнения.

Минимальная структура:

```txt
Walker {
    x: u16,
    y: u16,
    dir: Direction,
    load: u8,          // локальный заряд, который он несёт
    mode: WalkerMode,  // observe / choose / encode / runtime
    alive: bool,
}
```

## 7. Связь с L1 и L3

### Вход из L1

`L1` не пишет команды в `L2`.
`L1` подаёт:

- давление
- fingerprint
- локальную турбулентность
- насыщенность

То есть `CHAOS` не программирует `L2` напрямую.
Он возбуждает его поле.

### Выход в L3

`L2` не делает `MANIFEST`.
Он делает:

- устойчивый маршрут
- локально согласованную топологию
- operator-chain

И уже это схлопывается в `L3` как calm-структура.

## 8. Локальная запись (`p`-интуиция)

Из `Befunge` берётся не весь язык, а только один очень важный принцип:
пространство может переписываться локально.

Но запись разрешена только по топологии.

То есть:

- нет "пиши куда угодно"
- есть "измени локальную клетку или локальную связь, если это разрешено законом слоя"

Минимально:

```txt
fn topo_write(cell, neighborhood, walker) -> rewrite
```

Где `rewrite` может:

- сменить `op`
- ослабить или усилить связь
- сменить локальную фазу
- поставить `mark`

Но не может:

- телепортировать ходока
- писать произвольный байт в произвольную точку
- ломать тороидальную физику

## 9. Минимальный тик

Черновой цикл одного тика:

```txt
1. inject_from_L1()
2. diffuse_charge()
3. move_walkers()
4. apply_operator_rules()
5. apply_topo_rewrites()
6. detect_stable_routes()
7. emit_to_L3_if_ready()
```

## 10. Минимальные правила операторов

Не финальная истина, а стартовая интуиция.

### OBSERVE

- читает локальное давление
- увеличивает точность `mark`
- не пишет в `L3`

### CONNECT

- усиливает допустимые связи
- связывает 2 соседних активных направления

### DISSOLVE

- ослабляет или разрывает локальную связь
- сбрасывает stale-route

### CHOOSE

- схлопывает несколько возможных продолжений в одно
- повышает устойчивость выбранного направления

### ENCODE

- маркирует устойчивый маршрут как кандидат в кристалл
- готовит кусок для переноса в `L3`

### LOGIC

- фильтрует противоречивые локальные конфигурации
- может гасить невозможную связь

### RUNTIME

- удерживает маршрут живым несколько тиков
- не даёт ему распасться слишком рано

### CYCLE

- задаёт повторяемость локального обхода
- делает возможным возвращение в тот же контур

## 11. Что здесь уже похоже на правду

Похоже на правду:

- `L2` как тороидальная операторная сетка
- 8 операторов как узлы
- 4 ходока вместо одного
- локальная запись только по топологии
- двусторонняя мембрана между `CHAOS` и `CALM`

## 12. Что ещё не решено

Пока не решено:

- регулярная ли сетка или топология уже неравномерная
- фиксированы ли операторы клеток или они могут мигрировать
- как именно `L1` инжектит давление в `L2`
- какой критерий устойчивости достаточен для переноса в `L3`
- один ли тип ходоков нужен, или ролей должно быть несколько

## 13. Самый короткий тезис

`L2` сейчас видится как:

> тороидальный топологический клеточный автомат из 8 операторов,
> где несколько ходоков маршрутизируют и локально переписывают связи
> между `CHAOS` и `CALM`.

Это уже не `Befunge`.
Это `packet-boundary body`, на который `Befunge` только намекнул.
