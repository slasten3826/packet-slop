# Packet-Slop :: L2 Drain And PU Emergence

Этот документ фиксирует два важных вывода после первых живых `L2`-стендов.

## 1. Почему `L2` слишком быстро насыщается

Текущие `L2`-прототипы:

- действительно живут
- действительно строят `collapse`
- действительно делают `encode`
- действительно поднимают `calm`

Но они слишком быстро уходят в насыщение.

Причина не только в "слишком добрых правилах".

Причина глубже:

> у машины пока нет полноценного слива через `L3`

То есть сейчас картина такая:

- `L1` постоянно подаёт давление
- `L2` проводит и кристаллизует
- `L3` принимает результат
- но полноценного downstream-consumption ещё нет

Это похоже на ванну:

- вода наливается
- уровень растёт
- но нормального слива ещё нет

Отсюда:

- переполнение
- выравнивание активаций
- слишком лёгкое насыщение сетки

## 2. Закон перепада

Текущая модель слоя читается так:

- `L1` задаёт сильное напряжение
- `L3` задаёт более слабое встречное напряжение
- `L2` живёт на разности потенциалов между ними

Поток в целом идёт:

`L1 -> L2 -> L3`

Но внутри `L2` он может:

- ветвиться
- схлопываться
- кодироваться
- удерживаться

То есть `L2` — это не pipeline и не язык,
а boundary-network на перепаде напряжений.

## 3. Размер `L2` не должен задаваться руками

Следующий важный вывод:

`L2` не должен получать:

- ручной `width`
- ручной `height`
- ручной `PU`

Правильный режим:

- входом служат характеристики `L1`
  - `ring`
  - `CW`
- а `L2` сам выводит:
  - свой размер
  - свою ёмкость
  - свой `PU budget`

То есть:

```txt
L2_size = f(L1_ring, L1_CW)
PU_budget = g(L1_ring, L1_CW, L2_size)
```

Это уже реализовано в первом нейронном `L2`-стенде как черновой закон.

## 4. PU не должен быть просто числом сверху

`PU` не должен оставаться жёсткой константой.

Он должен проявляться как курс процесса.

То есть:

- маленький `L1` -> маленький `L2` -> дешёвый процесс
- большой `L1` -> большой `L2` -> дорогой процесс
- высокий `CW` удорожает проведение и кристаллизацию

Это похоже не на fixed budget,
а на курс валют или рыночную цену:

- цена работы рождается из состояния машины
- а не назначается вручную извне

## 5. Что уже видно по расходу PU

Текущий нейронный `L2` уже показывает правдоподобное распределение:

- `OBSERVE` — дешёвый
- `CHOOSE` — дороже
- `ENCODE` — самый дорогой
- `RUNTIME` — тоже дорогой, но обычно дешевле `ENCODE`

Это хорошо согласуется с packet-логикой:

- смотреть дёшево
- решать дороже
- кристаллизовать очень дорого
- коммитить и удерживать тоже дорого

## 6. Что это значит для следующих шагов

Следующий шаг по `L2` не в том,
чтобы просто "подкрутить пороги".

Следующий шаг в том,
чтобы честно достроить физику машины:

1. оставить связку `L1 -> L2 size/PU`
2. сохранить `PU` как emergent cost
3. ввести настоящий `L3 drain`
4. только потом заново смотреть на насыщение `L2`

Потому что без стока текущая картина неизбежно будет вести к переполнению.

## 7. Самый короткий тезис

`L2` сейчас уже жив,
но он ещё работает как:

> boundary-ванна без полноценного слива через `L3`.

Поэтому текущая задача не "починить нейросеть",
а достроить физику стока и цены процесса.
