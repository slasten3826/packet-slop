# Multi-L3 As Live RAM Hypothesis

Этот документ фиксирует промежуточную гипотезу.

Она пока не считается каноном.
К ней нужно будет вернуться позже.

## Исходная мысль

Если `L3` не один, а их несколько, и при этом цена жизни формы в `L3` падает с ростом числа `L3`, то третий слой начинает вести себя как живая оперативная память формы.

Не архив.
Не долгосрочная память.
А именно короткая рабочая память, где формы ещё живут, пока машина с ними работает.

## Что показал тестовый стенд

В тестовом `multi-L3` стенде:

- при `2 L3` формы всё ещё выгорают к концу окна
- при `3 L3` формы живут заметно дольше, но тоже умирают
- при `5 L3` формы доживают до конца короткого окна и умирают только на более длинном

Это значит:

- много `L3` не делает формы бессмертными
- но заметно удлиняет их жизнь
- и снижает цену удержания формы в рабочем пространстве

## Что это значит по смыслу

`L3` начинает выглядеть как:

- рабочее пространство
- живая short-term memory
- оперативка формы

Где:

- больше `L3` = больше одновременно живущих форм
- больше `L3` = ниже цена жизни одной формы
- больше `L3` = больше ширина рабочей памяти

## Почему это особенно важно для GPU

На `GPU` можно потенциально держать очень много `L3`-банок.

Если гипотеза верна, это даёт:

- дешёвую массовую life-memory
- много параллельных карманов жизни формы
- широкую оперативную память для фантомов
- недорогую short-term memory без классического transformer-RAG

То есть вместо retrieval из внешней базы можно получить среду, где формы ещё не извлекаются, а просто продолжают жить.

## Текущий статус

Это сильная идея, но пока рано тащить её в канон.

Сначала нужно:

- сделать более зрелый `L3`
- лучше понять `GPU`-экономику
- проверить, сохраняется ли этот эффект на более честном `L3 body`

## Самая короткая формула

`multi-L3` может оказаться не просто множеством рынков жизни формы, а дешёвой живой оперативной памятью формы, особенно хорошо ложащейся на `GPU`.
